Git的背景

有一份你正在编辑的大型word文档，中间一段你觉得写的不好，你想删除它，越看这一段你越觉得不顺眼。但是你害怕你的甲方或者老板觉得它写得好，所以你有一个问题：留着不顺眼，删除了又怕甲方觉得好；这个时候，同事A又来找你交接工作，他要改你的大型文档，这时候你总不能闲着，你要继续编辑这份文档。等同事A工作完成你还需要合并文档，查看修改的地方并合并修改，非常麻烦。怎么办？一种方法是备份文件，备份两份，一份删了中间那一段，自己编辑起来就很爽，另一份不删，留给老板看。至于工作的交接，只好手动完成，是不是很麻烦？另一个方法就是今天要介绍的工具–分布式版本控制系统-Git

Git的产生

Linux在流行初期，世界各地的志愿者都为其提交代码，为了让Linux变得更好，而提交的代码合并，纯粹靠linux本人手动合并，这是因为当时市场上流行的集中式版本控制系统CVS、SVN需要联网付费且速度慢，而且不开源，所以linux选择手动合并，后来linux在社区的要求下使用了一个版本控制系统，由于纠纷，这个版本控制系统也没有用多久，Git应运而生—使用C语言开发，为分布式版本控制系统，仅用两周时间。

集中式与分布式的区别

集中式：集中式的容易出现单点出了问题，全盘崩溃；
分布式：分布式不会因为某地分支出了问题就全盘崩溃

Git安装(Windows)

  • 进入官网默认选项安装即可

  • 安装完毕，进入git，设置自己的名字与邮箱：

    • git config --global user.name "xx"
    • git config --global user.email "xx@xx.com"

版本库/仓库操作

创建仓库

可以简单理解成一个目录，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。

仓库初始化

通过命令使这个文件夹成为git可以管理的仓库

git init

添加文件到仓库

  • 首先新建空白文件readme.txt到仓库目录
  • 然后执行git add readme.txt(git add用于将修改行为放入待提交状态)
  • 再执行git commit -m "说明" -m参数用于说明(git commit用于将待提交修改正式提交入仓库)
  • 执行git status用于检查是否有未提交的修改，是否有未放入待提交状态的修改以及，哪些文件被修改了

工作区与暂存区

工作区

电脑能看到的目录，如gitteset就是一个工作区

版本库(.git文件)
暂存区

git add xx.xx文件会先被保存到暂存区，可进行多次add命令

Header

指向master分支的指针

master分支
  • 进行git commit后，所有已经进行的git add操作都会被提交到master分支上，可以使用git status查看当前提交状态
    • 当我们只进行了add操作，查看提交状态会有
  • 当我们执行了commit操作后 可以看到，此时工作区(work tree)干净了

综合来看上述流程，其实就是：在工作区修改文件，修改好的文件通过add提交到暂存区，暂存区的文件再通过commit一次全部提交到master分支

对修改的操作

为什么Git比其他版本控制系统设计得优秀，因为Git跟踪并管理的是修改，而非文件。

什么是修改？
比如你新增了一行，这就是一个修改，删除了一行，也是一个修改，更改了某些字符，也是一个修改，删了一些又加了一些，也是一个修改，甚至创建一个新文件，也算一个修改。

管理修改
  • 我们对readme.txt添加内容123，
  • 然后git add readme.txt
  • 再次对readme.txt修改，添加内容asdf
  • 此时git commit ，提交成功
  • 再次使用git status查看当前提交状态，发现当前提交到master分支的只有123这次修改，为什么？
    • git的commit只对存于暂存区的修改进行提交,而第二次修改并没有add，也就是并没有存储到暂存区
  • 我们使用git diff HEAD -- readme.txt命令查看工作区和版本库里最新版本txt文件的区别，可以发现第二次修改确实没有保存进去
    • git diff？
  • 怎么办？
    • 再次git add readme.txt,git commit -m "xxx"即可
  • 输入git status,可以看到最后一次改动显示在了屏幕上
撤销修改
  • 对于修改内容的处理，我们可以使用命令git checkout --filename，意为将工作区所做的修改全部撤销，这里有两种情况
    • 情况1：文件自修改以后还没有放到暂存区，撤销修改后，工作区的文件就回到和版本库一模一样的状态
    • 情况2：文件已经添加到暂存区后，又做了修改，此时撤销修改就是回到添加到暂存区时（执行了add命令）的状态
  • 对于已经添加到暂存区的修改内容，我们可以使用命令git reset HEAD <filename>来撤销此次add，重新将修改从暂存区放回工作区
    • git reset命令既可以回退版本，也可以把暂存区的修改回退到工作区。当我们用HEAD时，表示最新的版本。
    • 放回工作区后，对工作区内容的撤销参考第一大点情况1

  • 对于已经提交到远程仓库的内容，没法子了。

以上内容基于廖雪峰2019年git文档，git对于上面的点进行了更新如下

  • 原来的git checkout 可以使用 git restore 代替
  • 原来的git reset HEAD 可以使用 git restore –staged 代替
  • 最新版的git提示都已经更换成了restore
  • 使用
    • 情况1：在工作区做了修改，并未添加到暂存区，想撤销工作区的修改，用 git restore file;
    • 情况2：在工作区做了修改，并用git add 添加到了暂存区，未提交；想撤销，分两步，1.先撤销暂存区的修改，用 git restore –staged file, 2.然后参考情况1撤销工作区的修改；
    • 情况3：在工作区做了修改，且git add git commit添加并提交了内容，想撤销本次提交，直接用 git reset –hard HEAD^回退版本，即可保证工作区，暂存区，版本库都是上次的内容
删除操作
  • 前提：你删除了本地仓库（工作区）中的文件
  • 现在要对版本库中的内容进行修改，对其进行修改有以下情形
    • 情形1：你确实需要删除版本库的文件
      • 这时你可以输入命令git rm xx.txt
      • 然后git commit，进行版本库中文件的删除
    • 情形2：你误删了工作区中的文件，现在想要恢复
      • 使用命令git checkout -- xx.txt
      • git checkout其实是用版本库里的版本替换工作区的版本，无论工作区是修改还是删除，都可以“一键还原”。
      • 命令git rm用于删除一个文件。如果一个文件已经被提交到版本库，那么你永远不用担心误删，但是要小心，你只能恢复文件到最新版本，你会丢失最近一次提交后你修改的内容。

添加远程仓库

什么是远程仓库？

存在一台中央服务器，里面存储了完整的仓库，每一个用户只需要登录自己的git并且下载中央服务器的仓库就可以实现对仓库的增删改查，同时也可以向中央服务器提交自己的更新并合并入中央服务器的仓库

  • 在进行后续的内容之前，我们需要配置SSH key
    • 在用户主目录下，看看有没有.ssh目录，如果有，再看看这个目录下有没有id_rsa和id_rsa.pub这两个文件，如果已经有了，可直接跳到下一步。如果没有，打开Git Bash，创建SSH Key：
    • `$ ssh-keygen -t rsa -C “youremail@example.com“
    • 可以在gitbash返回的目录里找到.ssh目录，里面有id_rsa和id_rsa.pub两个文件，这两个就是SSH Key的秘钥对，id_rsa是私钥，不能泄露出去，id_rsa.pub是公钥，可以告诉任何人。
